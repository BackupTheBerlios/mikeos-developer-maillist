Hi Takayoshi,

> The memory map of MikeOS 2.0 was :-
>
>        0000:8000 +----------------------+
>                  | Stack area   (SS)    |
>        0000:F000 +----------------------+
>
>        2000:0000 +----------------------+
>                  | User code    (CS,DS) |
>        2000:8000 +----------------------+
>                  | Kernel code  (CS)    |
>        2000:FFFF +----------------------+

That's right. I'm not sure if you saw on the list, but I've dropped
the idea of unreal mode and a radically new memory map. Instead, for
MikeOS 3.0, the memory map will be the same as above, with the second
two parts switched round.

So in the 2.9.x releases the memory map is:

        0000:8000 +----------------------+
                  | Stack area   (SS)    |
        0000:F000 +----------------------+

        2000:0000 +----------------------+
                  | Kernel code          |
        2000:8000 +----------------------+
                  | User (program) code  |
        2000:FFFF +----------------------+

Hence why programs are ORGed to 32768. I think this is a cleaner
design now that we don't have DOS compatibility -- does that make it
easier to update libmikeos?
