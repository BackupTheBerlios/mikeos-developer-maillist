On Mon, Nov 29, 2010 at 7:55 PM, Paulo Valongo
<paulovalongo at rocketmail.com> wrote:
>
> 1) Version 0.49 (where the mouse driver is used) still uses DOS INTs which the
> new versions don't
>    so any reference to DOS INT 33h (mouse) must not be used.

There are no references to int 33h or anything like that. The mouse
driver code is separate to the DOS stuff. (There's a handler for DOS
programs calling int 33h, but it just irets back.) The mouse driver
installs its own interrupt handler.

>  3) I don't quite know how the mouse cursor could be drawn in text mode and
> since all the os calls such as os_dialog_box, os_draw_background,
>     etc are all done in mode 03h (text) this sort of defeats the object of
> having a mouse in the first place

Yes, the two are mutually exclusive. The idea when I was making the
GUI branch was to eventually replace os_dialog_box with gui_dialog_box
etc. -- ie routines that would work in the 640x480x2 mode. I don't
know what'll happen if you call os_dialog_box in GUI mode; probably
all sorts of screen corruption!

> 4) Then there is the fact that many new motherboards don't even have PS2 ports
> but rely on BIOS doing some kind of translation from USB keyboards and mice to
> emulated PS2.
>    I wonder how well the PS2 port is emulated by BIOS as regards delays etc.

I had it working on a real laptop once. But the BIOS thing is a bit of
a problem. I was talking to some guys from Foxconn the other day, and
they were touting how much of their new kit is UEFI...
