Hi all,

I'd be interested in hearing opinions on the use of internal data
buffers inside system calls. For instance, in the latest
os_int_to_string implementation from Tslil (added to 3.2) the
resulting string is stored in a buffer inside the system call.

This is the case in many other MikeOS calls too -- they have their own
internal buffers.

The opposite approach to this is to have external programs define
their own data sections and pass the location of them to a system
call. This means that you have to throw another register into the mix,
but it makes the kernel slightly smaller.

So I wonder if anyone has any thoughts on this: should we try to keep
the use of data buffers inside the kernel system calls to a minimum,
and thereby put the onus on external programs to use them? Or is
everything fine and dandy as it is? :-)
