This is looking good -- thanks for all the work Paulo. It's great that
you've documented what you've done so far, so that others can help. If
anyone else is interested in having MikeOS on hard drives on day, it's
well worth checking out the .zip!

Now, the bit of copying bytes from one BPB to another is complex, and
I can see that introducing difficulties in the future. This is why I
think we should just have a single 16MB drive image with the relevant
data hard-coded into the bootloader BPB and MikeOS disk.asm routines.
Then running MikeOS 5 (or whatever) will be as simple as:

1) Plug in USB key
2) Run "dd if=mikeos.img of=/dev/usbkey" (or the Windows equivalent)

Sure, this wipes the USB key and leaves just 16MB of room, but for
MikeOS that's still enormous. Besides, any dedicated OS hacker should
have a spare USB key lying around, and you can pick up low capacity
ones for pennies now.

IMO, that's much simpler than telling people to format the key as
FAT16, then run some program to insert a bootloader which also
replaces bytes in the BPB with relevant info for that specific drive,
then run another script to copy the kernel and programs onto the drive
etc.

Let's go with the simple, always-the-same 16MB disk image approach,
which'll make it dead easy for people to try MikeOS. Paulo, if you get
the bootloader working on that disk image, I'll buy you a pint!
Somehow...
