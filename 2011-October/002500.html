Hi Mike,

>Now, the bit of copying bytes from one BPB to another is complex, and
>I can see that introducing difficulties in the future. This is why I
>think we should just have a single 16MB drive image with the relevant
>data hard-coded into the bootloader BPB and MikeOS disk.asm routines.

I agree.

>1) Plug in USB key
>2) Run "dd if=mikeos.img of=/dev/usbkey" (or the Windows equivalent)

I like the idea, however for safety's sake the following option must be
passed
to dd in Windows:

--filter=removable

It's also much trickier in Windows as dd reports/finds (removable) drives as
follows:

dd --list --filter=removable

and it will return something like this:

Removable media other then floppy. Block size = 512
size is xxxxxxxxx bytes
\\?\Device\Harddisk1\Partition0

The problem here is that the Harddisk number reported will vary depending
on how many physical HDDs (or partitions) the user has plus any other
removable drives that may be connected.
So it would make writing a script quite difficult as what would the
Harddisk number be?
It would vary from machine to machine.

If this can be overcome/standardised, then the fact that the USB key
will be limited to 16MB is really not a problem.
