On 10/22/08, Bob Moss <bobbyjmoss at googlemail.com> wrote:
>
> So you're wondering whether 32-bit offsets are ok for BIOS calls, which
> traditionally take 16-bit ones? You are also still remaining in real mode
> while doing this?

Correct. For instance, int13h / ah=03 (write sectors to disk) expects
the data location in ES:BX (16-bit). If it can't take ES:EBX (32-bit)
then my planned unreal mode memory map won't work.

> trapsing everywhere in real mode? So memory is not allocated dynamically,
> and memory protection is not really something you can implement in real
> mode?

Whether or not to have a memory manager vs letting apps grab what they
want is a kernel design issue -- hence why Tomasz and I are discussing
the possibility of memory management functions. But regardless of
design, in real mode you can't stop one app writing bytes over any
part of memory. In protected mode you use CPU features to limit
applications to their own memory area.

Hope that's clearer!
