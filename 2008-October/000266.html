On 10/3/08, Bob Moss <bobbyjmoss at googlemail.com> wrote:
>
> I'm assuming it wouldn't be overly difficult to move to FAT16, but I'd be
> interested to see you guys try to tackle FAT32 with a 16-bit kernel. As far
> as I can gather from my research it is actually possible, but it would be an
> absolute g*t to code in any language - let alone assembler!

It'd be a chore using 16-bit registers, yep, so 32-bit would make it
much simpler. But in fact, FAT32 is easier to work with in many
respects because the FAT entries fall neatly on byte boundaries
(rather than FAT12, in which you have to keep shifting or masking bits
depending on if it's an odd or even cluster).

You have to handle varying sectors per cluster (whereas it's always
one in FAT12) but that's no biggie. But otherwise it's largely the
same and easier to deal with thanks to the FAT entries being
double-words!
