On 10/22/08, Bob Moss <bobbyjmoss at googlemail.com> wrote:
>
> limitting factor for 32-bit (as you still need to send 16-bit instructions
> to the BIOS - it is an ancient piece of technology after all!).

Hang on a sec... I think you're confusing protected mode with 32-bit
addressing. With the 386 and upwards you can use 32-bit registers in
real mode; the issue here is whether 32-bit memory offsets are allowed
in BIOS calls (you don't "send instructions" to the BIOS).

> I do like the idea of memory segments by the way as it seems the most
> logical way to do it. But leaving it to the apps to allocate the memory is a
> bad idea (as it'll bite you in the bum later if you run a badly-written app
> or if someone wants to exploit a MikeOS system

And you're confusing segments with memory protection here :-) Whether
or not we use a single or multiple segments, and whether or not apps
allocate their own memory or via the kernel, we are still operating in
real mode so apps can stomp over any part of RAM. The only realistic
way around this is with protected mode, which would leave a large
chunk of the MikeOS source code dealing with floppy disk drivers and
the like, so that's not going to happen.
