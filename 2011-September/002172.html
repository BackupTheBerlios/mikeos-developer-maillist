> I learn that using some such as $a or $d (case sensitive) would cause
> trouble, so I avoided them.

Given that the BASIC interpreter works on ASCII values for the string
variables as mentioned, you're better off using upper-case letters:

http://www.asciitable.com

eg 'A' is closer to the 0 - 9 characters than 'a'. Now that the string
storage will be at the end of the kernel for 4.2, there's lots of
space free, but it gets tight towards the end. For instance, the
string variable '$z': z = 122 in ASCII, subtract 49, and then multiply
the result by 128 = 9344 bytes offset. So with the kernel at around
16K and the disk buffer at 24K, using variables like $x, $y and $z
intrudes in the disk buffer space.

Using '$Z' (capital) results in 5376 offset which is safely in that free RAM.

> If I have to make a call on this one (call 65515?) I'd say do, you don't have to
> do all possible characters, just to case insensitive letters. Most of the time
> I don't use the full 128 characters anyway

Righto, sounds like an idea. I won't change anything for 4.2 now --
there are enough changes in there already, so I just want to focus on
making it stable. But for 4.3 I'd like to focus a lot on BASIC, so we
could investigate changes like 64-char strings, using $0 - 9 and $A -
Z (with the interpreter dealing with the case itself). This could
potentially break things like Bazza's projects, so we'd give this lots
of testing. But that's a while off yet.
